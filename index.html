01Compute & Serverless

🟧 AWS Lambda — Kiến thức cốt lõi

Lambda chạy code không cần quản lý server. Mỗi lần invocation chạy trong một execution environment (sandbox MicroVM). Vòng đời gồm 2 pha quan trọng:

  • Init phase (allocation time): tải code, load layers, chạy code ở global scope, khởi tạo SDK client, mở DB connection. Khi có Provisioned Concurrency, pha này chạy TRƯỚC khi request đến → không tính vào duration của handler.
  • Invoke phase: chạy handler. Cold start = lần đầu khởi tạo môi trường mới. Warm start = tái sử dụng môi trường sẵn có.

❄️ Cold start — các cách giảm

Kỹ thuậtHiệu quảLưu ý
Provisioned ConcurrencyGiữ sẵn N execution environment đã chạy initChỉ gán cho version cụ thể hoặc alias, KHÔNG gán được cho $LATEST. Phải publish version mới gán được.
Reserved Concurrency❌ Không giảm cold startChỉ giới hạn số concurrent execution (quota).

Mẹo thi: Câu hỏi nhắc "predictable invocation duration" + "init phải chạy ở allocation time" → chọn Provisioned Concurrency kết hợp publish version. Đừng chọn $LATEST!

🌉 Lambda VPC access

  • Mặc định Lambda KHÔNG nằm trong VPC → không truy cập được resource private (RDS, EC2 private subnet).
  • Để truy cập: attach Lambda vào VPC + private subnets + security group. Lambda sẽ tạo Hyperplane ENI trong subnet.
  • ❌ Sai Dùng NAT Gateway để Lambda nối với resource trong VPC (NAT chỉ cho outbound Internet).
  • ❌ Sai Dùng VPC Endpoint để Lambda nối với EC2 (Endpoint dành cho AWS service như S3, DynamoDB).
  • ❌ Sai Dùng VPN cho Lambda ↔ EC2 trong cùng VPC.

🔌 Lambda + RDS / Aurora

  • Surge traffic + nhiều Lambda mở connection → dùng RDS Proxy để pool connection, tránh vượt max_connections.
  • Best practice: mở DB connection ở global scope (ngoài handler) để reuse giữa các warm invocation. Đây là pattern cải thiện latency lớn nhất khi Lambda gọi DB thường xuyên.

⏱️ Lambda timeout vs SQS Visibility Timeout

Khi Lambda được trigger bởi SQS, nếu xử lý lâu hơn Visibility Timeout của queue → message sẽ tái xuất hiện và bị xử lý trùng lặp.
Quy tắc vàng: SQS Visibility Timeout ≥ Lambda timeout + buffer, và phải đủ dài để cover cả call ngoài (vd 60s gọi API ML).
Tăng Lambda timeout/memory KHÔNG sửa được trùng lặp — phải tăng Visibility Timeout!

🎯 Lambda Destinations (async invocation)

Khi Lambda invoke async (vd từ SNS, S3, EventBridge), có thể cấu hình:

  • Success destination: gửi kết quả khi thành công
  • Failure destination: gửi event khi lỗi (sau retry)

Target hỗ trợ: SQS / SNS / EventBridge / Lambda.

Destinations khác DLQ:
  • Destinations: cả success + failure, có metadata đầy đủ.
  • DLQ: chỉ failure sau khi retry hết.
Muốn log MỌI invocation → dùng cả 2 destination (success + failure) cùng trỏ vào 1 SQS.

🎚️ Lambda Alias + Weighted Routing (canary / staging)

  • Alias trỏ tới 1 hoặc 2 version với % traffic. Cho phép canary: ví dụ 90% prod, 10% new version.
  • SDK / API Gateway gọi alias staging → Lambda tự split traffic.
  • ❌ Không gán weight cho version trực tiếp — chỉ alias mới có weight.

💾 Lambda /tmp Ephemeral Storage

  • Mỗi execution environment có thư mục /tmp, mặc định 512 MB, tăng tối đa 10 GB.
  • Tốt cho workload đọc đi đọc lại một file lớn (vd: download video từ S3, xử lý nhiều pass) → đọc local nhanh hơn rất nhiều so với gọi S3 GetObject lặp lại.
  • ❌ Lambda KHÔNG gắn EBS, KHÔNG gắn ENA — chỉ có /tmp hoặc mount EFS.

🟪 AWS CodeDeploy — Deployment Strategies

  • Canary: shift 1 phần nhỏ trước (vd 10% trong 5 phút), nếu OK → roll 100% cùng một lúc → phù hợp "thử nhóm nhỏ rồi deploy đồng loạt cho all users".
  • Linear: shift theo % đều đặn (vd 10% mỗi 10 phút).
  • All-at-once: deploy hết cùng lúc.
  • In-place vs Blue/Green: in-place update trực tiếp instance đang chạy; blue/green tạo môi trường mới và switch.
Lỗi IAM_ROLE_PERMISSIONS trong CodeDeploy → gắn policy AWSCodeDeployRole (cho EC2/On-Prem) hoặc AWSCodeDeployRoleForECS (cho ECS). KHÔNG nhầm với agent issue!

🟫 Amazon ECS Fargate

  • Fargate không quản lý EC2 host. Container chạy trên hạ tầng AWS managed.
  • Shared persistent storage giữa nhiều task → dùng Amazon EFS qua efsVolumeConfiguration + mountPoints trong task definition.
  • ❌ EBS gắn vào EC2 instance, không gắn tự nhiên cho Fargate task.
  • ❌ DockerVolumeConfiguration chỉ hỗ trợ EC2 launch type, KHÔNG hỗ trợ Fargate.
  • ECS Task IAM Role: bật ECS_ENABLE_TASK_IAM_ROLE=true → cấp quyền riêng cho từng task → least privilege. Khác với EC2 instance profile role (cho ECS agent).

02Storage — S3, EBS, EFS, ElastiCache

🟧 Amazon S3 — Kiến thức cốt lõi

📁 S3 Versioning + Lifecycle

  • Khi bật Versioning, mỗi PutObject tạo một version mới. Object cũ thành noncurrent version.
  • Để giới hạn số version giữ lại (vd: current + 1 trước), dùng S3 Lifecycle Rule với action NoncurrentVersionExpiration + tham số NewerNoncurrentVersions.
  • ❌ Bucket policy chỉ quản lý access, KHÔNG quản lý retention.
  • ❌ Object Lock = WORM (compliance), không dùng để giới hạn version count.
  • ❌ Suspend Versioning + sửa code app → không thực tế.

🔐 S3 Encryption — Phân biệt rõ

LoạiAi quản lý keyĐặc điểmUse case
SSE-S3S3 tự quản lý (AES-256)Đơn giản nhất, không audit qua CloudTrailYêu cầu "chặn third-party" cơ bản
SSE-KMSAWS KMS (Customer Master Key)Audit qua CloudTrail, kiểm soát quyền kms:DecryptCần kiểm soát chi tiết, compliance
SSE-CCustomer cung cấp keyS3 không lưu key, key đi kèm mỗi requestTự quản lý key hoàn toàn
DSSE-KMS2 lớp mã hóa, KMSCompliance cực caoWorkload nhạy cảm đặc biệt
Client-sideClient mã hóa trước khi uploadDùng GenerateDataKey + envelope encryption"Encrypt BEFORE upload"
Mẫu câu hỏi phân biệt:
  • "Encrypt BEFORE upload" → Client-side với kms:GenerateDataKey. Flow: Lambda gọi GenerateDataKey → nhận plaintext key + CiphertextBlob → mã hóa file bằng plaintext key → upload kèm CiphertextBlob trong metadata.
  • "Encrypt AT REST" → SSE-S3 / SSE-KMS (đơn giản, không cần đụng code).
  • "Chặn third-party basic" → SSE-S3 là đủ (không cần SSE-KMS phức tạp).

🔒 In-transit Encryption (S3)

HTTPS bảo mật kênh truyền. Để ENFORCE HTTPS cho bucket: dùng bucket policy với điều kiện aws:SecureTransport = false để Deny request HTTP.

{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": ["arn:aws:s3:::bucket","arn:aws:s3:::bucket/*"],
  "Condition": {"Bool": {"aws:SecureTransport": "false"}}
}
  • ❌ Cài cert trên EC2, dùng VPC Endpoint, SSE-KMS... đều KHÔNG enforce HTTPS phía S3.

🪄 S3 Object Lambda

  • Cho phép biến đổi data on-the-fly khi client gọi GetObject thông qua S3 Object Lambda Access Point.
  • Use case điển hình: redact PII, resize ảnh, convert format — KHÔNG cần lưu nhiều bản copy hay refactor consumer.
  • Khác biệt quan trọng: Mã hóa (SSE) KHÔNG che PII — decrypt xong vẫn thấy. Muốn che PII phải dùng transformation thật sự.

🪣 S3 + SQS Pattern

SQS message tối đa 256 KB. Với file 10 MB:
User upload→ S3 (Standard)→ Gửi S3 location (key/URL) vào SQS→ Consumer xử lý
Không bao giờ nhúng nội dung file lớn vào message SQS.

🟦 Amazon EFS

  • File system NFS managed, share giữa nhiều EC2 / ECS task.
  • Hỗ trợ Fargate mount qua task definition (efsVolumeConfiguration).
  • Latency cao hơn nhiều so với /tmp local — phù hợp shared persistent storage, không cho I/O cực nặng.

🟫 Amazon EBS

  • Block storage gắn vào 1 EC2 instance (gp3 default).
  • Không share trong cùng Region (trừ io2 Block Express multi-attach giới hạn).
  • Không dùng cho Lambda hoặc Fargate shared storage.

🟨 Amazon ElastiCache

  • In-memory cache, latency microsecond — nhanh nhất trong nhóm cache.
  • Memcached: đơn giản, multi-thread, không persistence → phù hợp session cache.
  • Redis: persistence, pub/sub, replication, cluster mode.
"Lowest-latency session store" → luôn chọn ElastiCache.
So sánh: DynamoDB / RDS đều có latency ms (chậm hơn), shared filesystem chậm hơn nữa.
Pattern stateless web tier: web server EC2 (Auto Scaling) + session lưu ElastiCache + data lưu RDS.

  • In-memory cache cho DynamoDB → giảm latency đọc từ ms xuống microsecond.
  • ❌ Chỉ tăng tốc đọc, KHÔNG phát stream/event → không thay thế Streams được.

🟦 Amazon RDS / Aurora

  • Aurora MySQL/PostgreSQL: managed, scale storage tự động, replica nhanh.
  • Khi có nhiều Lambda concurrent connect → dùng RDS Proxy (connection pooling, IAM auth, lấy credential từ Secrets Manager).
  • Connection bảo mật: dùng Secrets Manager rotation; Lambda fetch credential tại runtime (KHÔNG dùng env var vì env var bị đóng băng lúc init).
  • Encryption at-rest của RDS không tự giải mã dữ liệu mức ứng dụng (vd ciphertext từ CloudFront Field-Level Encryption).

🟦 Amazon CloudFront

  • CDN toàn cầu, cache tại edge, giảm latency.
  • Field-Level Encryption (FLE): mã hóa các field nhạy cảm tại edge bằng RSA public key (từ KMS). Origin/ứng dụng PHẢI DECRYPT bằng private key trước khi lưu DB.
Bug điển hình: Bật CloudFront FLE → DB lưu ciphertext thay vì plaintext.
Nguyên nhân: Lambda/app ở origin không decrypt.
Fix: Thêm Lambda function dùng KMS key để decrypt các data field TRƯỚC KHI ghi vào DB.
SSE-KMS hay HTTPS đều không giải quyết được — chúng chỉ mã hóa kênh truyền / tầng lưu trữ, không decrypt ciphertext của FLE.

🌍 Headers đặc biệt cho content theo địa lý + ngôn ngữ

  • CloudFront-Viewer-Country: mã quốc gia của user (tự động).
  • Accept-Language: ngôn ngữ trình duyệt user.
User→ CloudFront (edge)→ Forward 2 headers→ Origin xử lý logic→ Trả content phù hợp

05Security & Identity

🟧 IAM — Least Privilege

  • Luôn cấp đúng quyền cần thiết, scope theo Resource ARN cụ thể.
  • DynamoDB read actions: GetItem, BatchGetItem, Query, Scan.
  • DynamoDB write actions: PutItem, UpdateItem, DeleteItem, BatchWriteItem.
  • Trust policy với aws:SourceArn = ARN chính xác của resource (vd state machine của Step Functions) → chỉ resource đó mới assume được role.
  • ❌ Tránh dùng "*" trong Resource hoặc StringNotEquals sai ngữ cảnh → cấp quá rộng.
"Condition": {
  "ArnLike": {
    "aws:SourceArn": "arn:aws:states:ap-south-1:111111111111:stateMachine:myStateMachine"
  }
}

🟦 AWS KMS

  • Customer managed key: tự quản lý key policy → cho phép cross-account access. AWS managed key KHÔNG sửa được policy.
  • Cross-account secret access: cần kết hợp:
    • IAM policy (Secrets Manager) ở app account.
    • KMS key policy (kms:Decrypt, kms:DescribeKey) ở key owner account.
  • GenerateDataKey: trả về plaintext data key + CiphertextBlob → dùng cho envelope / client-side encryption.
    • Plaintext key mã hóa data.
    • CiphertextBlob lưu kèm object để decrypt sau (qua KMS).
  • ❌ KMS keys KHÔNG replicate cross-account; phải tạo key mới và cấp policy.

🟪 AWS Secrets Manager

  • Lưu credential DB, API key, connection string. Hỗ trợ automatic rotation.
  • Lambda fetch tại runtime để luôn dùng giá trị mới nhất (KHÔNG nhúng qua env var — env var đóng băng lúc init).
  • Lambda Extension của Secrets Manager: caching ở môi trường runtime.
Mẹo: Yêu cầu "credential update tự động, không sửa code/config" → Secrets Manager + fetch runtime. Loại Parameter Store reference qua env var.

🟨 AWS Systems Manager Parameter Store

  • Lưu config và secret (SecureString = mã hóa qua KMS).
  • Rẻ hơn Secrets Manager, nhưng KHÔNG có rotation built-in cho DB credential.

🟫 AWS AppConfig

  • Quản lý feature flags + config dynamic.
  • AppConfig Agent trên EC2 / Lambda Extension poll định kỳ, cache local, app đọc qua localhost endpoint → overhead tối thiểu, rollback an toàn.
AppConfig (control plane)→ Agent poll trên EC2/Lambda→ App đọc localhost

06Messaging & Integration

🟧 Amazon SNS

  • Pub/Sub topic, fan-out tin nhắn tới nhiều subscriber (SQS, Lambda, HTTPS, SMS, email).
  • Standard topic: best-effort ordering, at-least-once.
  • FIFO topic ⭐: ordering (MessageGroupId) + deduplication. Phù hợp nghiệp vụ cần đảm bảo thứ tự và không trùng (vd: SMS xác nhận giao dịch chứng khoán).
  • Subscription filter policies: lọc message tại subscription level (theo message attribute). Tốt nhất cho fan-out có điều kiện — KHÔNG cần lọc trong code consumer.
Pattern fan-out chuẩn:
S3 event→ SNS topic→ SQS A (filter: dept=A) + SQS B (filter: dept=B)
KHÔNG cần sửa Lambda code → ít vận hành.

🟦 Amazon SQS

LoạiĐặc điểmKhi dùng
StandardThroughput cao, ordering best-effort, at-least-onceWorkload chấp nhận trùng/sai thứ tự
FIFOOrdering + exactly-once, throughput thấp hơnNghiệp vụ cần thứ tự, không trùng

⚙️ Các khái niệm quan trọng

  • Visibility Timeout: thời gian message bị "ẩn" sau khi consumer đọc. Lambda timeout phải ≤ visibility timeout - buffer.
  • Long polling (WaitTimeSeconds 1–20): giảm empty receive, tiết kiệm chi phí.
  • Batching: ReceiveMessage có thể lấy tối đa 10 message/lần → giảm request, giảm chi phí.
  • DLQ (Dead-Letter Queue): nhận message thất bại sau khi hết maxReceiveCount; chỉ log lỗi, KHÔNG log success.
  • SQS message tối đa 256 KB; với payload lớn dùng SQS Extended Client + S3.
Cost-effective SQS Standard: Long polling + Batching = chuẩn.

🟪 EventBridge / Pipes / Scheduler

  • EventBridge Rules: pattern match event từ AWS service hoặc custom → target (Lambda, SNS, SQS, Step Functions...).
  • EventBridge Scheduler: lịch hẹn giờ (cron/rate) — chỉ để invoke, KHÔNG điều phối workflow theo phụ thuộc.
  • EventBridge Pipes: pipe source → filter → enrichment → target. KHÔNG gửi SMS trực tiếp.

🟨 AWS Step Functions

  • Orchestration workflow (Amazon States Language).
  • Phù hợp khi cần chạy nhiều Lambda theo thứ tự, với retry/error handling, parallel, choice...
  • So sánh: SNS/SQS KHÔNG đảm bảo thứ tự thực thi giữa nhiều function khác nhau theo chain logic.
Order received→ Validate Lambda→ Payment Lambda→ Inventory Lambda→ Notification Lambda

🟫 Amazon Kinesis Data Firehose

  • Streaming ETL tới S3, Redshift, OpenSearch, HTTP endpoint.
  • Hỗ trợ data transformation via Lambda: biến đổi/redact PII trước khi ghi xuống destination.
  • ❌ KHÔNG gửi SMS trực tiếp, KHÔNG target là Step Functions/EC2.
Producer→ Firehose→ Lambda redact PII→ S3 (sạch)

07CI/CD & Infrastructure as Code

🟧 AWS CodePipeline

Orchestration pipeline: Source → Build → Test → Deploy. Tích hợp CodeCommit/GitHub/S3 (source), CodeBuild (build/test), CodeDeploy, CloudFormation, Step Functions.

🟦 AWS CodeBuild

Managed build server. Chạy script tùy ý theo buildspec.yml: build, unit test, integration test, package artifact.

🟪 AWS CodeDeploy

  • Deploy app lên EC2 / On-Prem, Lambda, ECS.
  • Service role cần policy AWSCodeDeployRole (EC2/On-Prem) hoặc AWSCodeDeployRoleForECS.
  • Lỗi IAM_ROLE_PERMISSIONS → gắn đúng policy tương ứng.
  • Deployment config: Canary, Linear, AllAtOnce → trả về traffic shifting cho Lambda alias.

🟨 AWS CloudFormation

  • IaC khai báo bằng JSON/YAML.
  • DeletionPolicy: mặc định Delete. Có thể đặt Retain hoặc Snapshot để bảo toàn resource khi update/delete stack.
  • ⚠️ Đổi tên resource (logical ID hoặc Name property) trong template = CloudFormation xem là resource mới → tạo mới + xóa cũ (theo DeletionPolicy).
  • Change sets: preview thay đổi trước khi apply, KHÔNG phải công cụ test code local.
  • StackSets ⭐: deploy stack trên nhiều account/region một lần → phù hợp multi-account.
Bug điển hình: Đổi tên DynamoDB table trong template + DeletionPolicy mặc định → CloudFormation TẠO MỚI bảng + XÓA bảng cũ. Mất data!
Phòng tránh: Đặt DeletionPolicy: Retain cho resource quan trọng.

🟫 AWS SAM (Serverless Application Model)

  • Extension của CloudFormation cho serverless.
  • sam local invoke / sam local start-api: test Lambda + API Gateway local.
  • Tích hợp CodePipeline: dùng SAM CLI commands trong các stage để build, package, deploy.

🟧 Multi-account CI/CD Pattern

Provision test infra qua nhiều AWS account + chạy integration test:
CodePipeline→ CloudFormation StackSets (multi-account)→ CodeBuild chạy tests
Đây là pattern chuẩn, ít vận hành nhất.

🟦 AWS X-Ray

  • Distributed tracing. Bật trên Lambda (active tracing) hoặc SDK instrumentation.
  • Service map: các node (service) và edge (line giữa chúng).
  • Click EDGE = latency BAO GỒM network giữa 2 service (Lambda → S3).
  • Click NODE = chỉ đo latency của service đó (không gồm network).
  • Active tracing cho SNS: đo rate + latency publish → deliver, đặc biệt hữu ích cho FIFO topic.

🟪 VPC Flow Logs

  • Ghi IP traffic in/out của ENI.
  • Dùng để kiểm tra request có tới VPC chưa, traffic bị accept/reject.
  • Use case điển hình: debug ALB timeout từ laptop — xem Flow Log để biết packet có đến không.

🟨 AWS Config

  • Theo dõi config và compliance của resource.
  • Managed rules ví dụ: acm-certificate-expiration-check, s3-bucket-public-read-prohibited.
  • Kết hợp EventBridge rule (Config Rules Compliance Change) + SNS = cảnh báo tự động.

🟫 AWS CloudTrail

  • Ghi API call trên account (audit).
  • ❌ KHÔNG ghi performance/latency của service.

10Cheat-sheet — Cách các service kết nối với nhau

  • S3 → Lambda: S3 Event Notification (s3:ObjectCreated:*) hoặc qua SNS/SQS/EventBridge.
  • S3 → SNS → nhiều SQS: fan-out với subscription filter policy.
  • API Gateway → Lambda: REST/HTTP API tích hợp Lambda proxy; HTTP API hỗ trợ JWT authorizer built-in.
  • Lambda → RDS private: Lambda attach VPC + private subnet + SG; thêm RDS Proxy để pool.
  • Lambda async → destination: Success/Failure destination = SQS, SNS, EventBridge, Lambda.
  • SQS → Lambda: event source mapping; Visibility timeout phải ≥ Lambda timeout.
  • SNS FIFO → SMS/SQS/Lambda: ordering + dedup, hỗ trợ fan-out.
  • DynamoDB Streams → Lambda: NEW_IMAGE / OLD_IMAGE / NEW_AND_OLD / KEYS_ONLY.
  • Kinesis Firehose → Lambda transform → S3: redact/transform trước khi lưu.
  • CloudFront → Origin (ALB/S3/Lambda function URL): forward header (Accept-Language, CloudFront-Viewer-Country) để origin trả content phù hợp.
  • Step Functions → Lambda/ECS/SNS/SQS: orchestration tuần tự/song song.
  • CodePipeline → CodeBuild → CloudFormation/StackSets/SAM/CodeDeploy: CI/CD chuẩn.
  • Config → EventBridge → SNS: phát hiện compliance change + thông báo.
  • Cognito User Pool → API Gateway authorizer: bảo vệ backend API bằng JWT.
  • Cognito Identity Pool → STS → S3/DynamoDB: cấp temporary AWS credentials cho client.
  • KMS GenerateDataKey → Client encrypt → S3 PutObject: client-side envelope encryption.
  • Secrets Manager → Lambda fetch runtime → RDS: credential luôn mới sau rotation.
  • AppConfig → AppConfig Agent (sidecar) → EC2/Lambda localhost: feature flag polling.
  • CloudWatch agent (on-prem) → CloudWatch Logs: tập trung log từ máy on-prem.
  • X-Ray active tracing → Service map: phân tích latency, click edge để gồm network.
Nguyên tắc vàng khi thi DVA-C02:
  • "Least operational overhead" → ưu tiên managed service, built-in feature (vd JWT authorizer, EMF, AppConfig Agent).
  • "Least privilege" → cấp đúng quyền, scope ARN cụ thể, dùng role thay vì user.
  • "Most secure" → temporary credentials (IAM role) thay vì long-term keys; KMS thay vì hardcode.
  • "Lowest latency" → ElastiCache cho cache, Regional endpoint cho STS, /tmp cho Lambda local.
  • "Cost-effective" → S3 Standard cho hot data, Long polling cho SQS, Lambda Provisioned chỉ khi cần.
